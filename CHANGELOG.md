- **BandwidthTracker Emby attribution wiring (bd-gih6d, epic bd-2cenq).** Wires the bd-6802c cross-ref resolver into `backend/bandwidth_tracker.py` so `session_telemetry` rows carry the resolved Emby viewer's identity. **New method `BandwidthTracker._resolve_emby_attributions(telemetry_channel_snapshot, provider_by_channel)`** is called from `_collect_stats` after the bd-skqln.14 provider resolver runs and BEFORE `_write_session_telemetry`; it builds a sparse `{(channel_uuid, client_ip): EmbyAttribution}` map by iterating the per-channel snapshot's `(channel, ip)` pairs and awaiting `services.emby_resolver.resolve_emby_user(client_ip, provider_resolution.stream_name)` for each. **Settings gate optimization** reads `settings.emby_enabled` at the top via a local `from config import get_settings` and short-circuits with an empty map when False — the cache and resolver already short-circuit on disabled, but skipping the per-(channel, ip) loop, the per-call settings access, and the resolver function-call overhead on the common disabled-Emby install is the cheap optimization the bead spec explicitly calls out. **Stream-name skip** also avoids the resolver call when `provider_resolution.stream_name` is None (the cache match is name-based so the resolver would short-circuit inside anyway — skipping at the call site saves the function-call overhead). **Failure handling** wraps each resolver call AND the settings read in try/except: any exception (resolver is documented to never raise, but a future refactor or unforeseen runtime fault like cache failure or settings access raise must not block telemetry writes) collapses to a single rate-limited `[BANDWIDTH] [EMBY]` WARN via the new module-level `_log_emby_resolver_failure(exc)` helper. The rate-limit window is **60 seconds** anchored on a module-level `time.monotonic()` timestamp (`_emby_resolver_last_warn_at`) so multiple resolver failures within one busy poll cycle — and across poll cycles inside the window — collapse to ONE WARN line, preserving the operational signal-to-noise ratio. The module-level (not instance-level) timestamp means multi-tracker test seams share the same clock and the failed-resolver row still writes with NULL Emby columns. **`_write_session_telemetry` signature extended** with a keyword-only `emby_attributions: Optional[dict[tuple[str, str], EmbyAttribution]] = None` parameter (default empty for back-compat with older test seams); the per-row construction looks up `emby_attributions.get((channel_uuid, ip))` and passes `attribution.user_id` / `attribution.user_name` (or `None` when missing) to the `SessionTelemetry(...)` constructor's new `emby_user_id` / `emby_user_name` kwargs from migration 0016. Both columns move together — the dataclass returns them atomically, and a future split into a separate `emby_users` table can migrate this writer in one place. **The hot-path discipline mandate** "Telemetry write must NEVER block on Emby failure" is enforced by the try/except wrapper AND by keeping the resolver call entirely OUTSIDE the SQLAlchemy session that owns the write — a resolver raise cannot poison the session commit, and the row still lands with NULL Emby columns regardless of which IP / channel pair triggered the failure. **Tests:** 7 new tests in `backend/tests/unit/test_bandwidth_tracker_emby.py` lock the full behavioral matrix — (1) resolver returns `EmbyAttribution` → row has both Emby columns populated; (2) resolver returns `None` → row has both Emby columns NULL; (3) `settings.emby_enabled = False` → resolver mock asserts NOT awaited and rows have both Emby columns NULL; (4) resolver raises `EmbyClientError` → row STILL writes with NULL Emby columns AND exactly ONE `[BANDWIDTH] [EMBY]` WARN is emitted; (5) rate-limit guard: 6 consecutive failures across one busy poll cycle collapse to exactly ONE WARN; (6) end-to-end realistic scenario: Dispatcharr poll with one client whose IP is the Emby server's IP + matching cached Emby session → telemetry row carries the resolved Emby attribution alongside the bd-skqln/bd-kh23e Stats v2 fields (`channel_id`, `stream_id`, `stream_name`); (7) sparse-map per-IP attribution: two clients on one channel where only one resolves → matching client's row gets the attribution, non-matching client's row keeps NULL Emby columns without spilling another row's attribution. - **Emby cross-ref resolver: ECM stream → Emby user (bd-6802c, epic bd-2cenq).** New `backend/services/emby_resolver.py` answers the load-bearing user-attribution question for the v0.17.x Emby epic: given an ECM stream session's client IP and stream name, which Emby user — if any — is the actual viewer? Without this cross-reference Stats > User Watch Time collapses every Emby viewer into a single proxy-IP identity. **Module-level `async resolve_emby_user(ecm_session_ip, ecm_stream_name) -> EmbyAttribution | None`** is the only public entry point and consumes the bd-gpeot `get_cached_emby_sessions()` cache (NEVER instantiates `EmbyClient` directly — going around the cache would defeat the thundering-herd guard, stale-fallback, and settings gate). **Algorithm:** (1) extract Emby server IP from `settings.emby_base_url` via `urlparse(...).hostname` — IP literals (`http://192.168.1.10:8096`) are used verbatim, hostnames (`https://emby.local:8920`) are resolved once via `socket.gethostbyname` and cached module-level (a successful resolution is cached as the IP string, a `socket.gaierror` failure is cached as the `_UNRESOLVED` sentinel so the WARN log fires exactly once per process and subsequent polls short-circuit cheaply); IP-literal detection uses `socket.inet_pton` for both AF_INET and AF_INET6 rather than naive dot-counting; (2) if the ECM session IP does not equal the Emby server IP, return `None` BEFORE the cache call — every poll cycle hits this branch for every non-Emby-mediated session, so the short-circuit is the load-bearing optimization; (3) fetch cached Emby sessions (the cache transparently handles the disabled-Emby case by returning `[]`, so the resolver does not need to inspect `emby_enabled`); (4) for each session, score `ecm_stream_name` against BOTH `now_playing_item_name` AND `now_playing_channel_name` (live-TV sessions populate channel_name, VOD populates only item_name, idle sessions populate neither) — exact match after NFC + lowercase + strip normalization wins immediately, else RapidFuzz `fuzz.token_set_ratio / 100` against the bead-mandated `FUZZY_MATCH_THRESHOLD = 0.85`; (5) zero matches → `None`, one match → `EmbyAttribution(user_id, user_name)`, multiple matches (rare — same channel on multiple Emby clients) → tie-break on most-recent `last_activity_date` (Emby ISO timestamps sort lexicographically the same as chronologically; `None` timestamps lose to any populated value via the `or ""` fallback in the key function). **`EmbyAttribution` is a frozen dataclass** with `user_id` and `user_name` so callers cannot mutate the attribution between the resolver returning it and the BandwidthTracker writing it to `session_telemetry.emby_user_id` / `emby_user_name`. **Never raises** — the BandwidthTracker poll loop calls this on every active session every ~5 seconds, so defensive failures (empty URL, malformed URL, DNS failure) return `None` after a one-shot WARN rather than propagating up. **Normalization is duplicated from `dedup_matcher._normalize`** rather than imported — the two services answer different questions with different threshold policies, and coupling them via a shared helper would couple policies that may evolve independently; the cost of a one-liner duplication is lower than the cost of forcing a shared module on two unrelated callers. **Logging** uses the `[EMBY]` prefix with lazy `%` formatting per `backend/CLAUDE.md`; resolver decisions log at DEBUG (don't spam INFO on every poll), DNS failure logs at WARN exactly once via the cache sentinel. **Settings-disabled short-circuit** is implicitly handled by the cache layer returning `[]`; no explicit `emby_enabled` check in the resolver. **Tests:** 17 new tests in `backend/tests/services/test_emby_resolver.py` lock the full behavioral matrix — IP mismatch returns `None` without a cache call (the load-bearing optimization), exact match on item_name returns attribution, case-insensitive match ("cnn" matches "CNN"), channel_name match when item_name is None (live-TV path), fuzzy match above 0.85 threshold returns attribution, fuzzy match below 0.85 returns None, empty cache returns None (Emby disabled / idle), all-idle sessions return None, multi-match tiebreak by most-recent `last_activity_date`, populated timestamp beats None timestamp, hostname-based base_url resolves to matching IP and attributes, hostname resolving to non-matching IP short-circuits before cache, DNS failure logs WARN with `[EMBY]` prefix and hostname and returns None gracefully, DNS resolution result is cached (3 successive calls fire `gethostbyname` exactly once), empty base_url returns None defensively, malformed base_url returns None defensively. The resolver is now ready for bd-gih6d's BandwidthTracker enrichment wiring; this bead intentionally does NOT touch BandwidthTracker (bd-gih6d) or the Stats endpoints (bd-fm23o). ([bd-6802c])

- **Emby session cache with TTL + thundering-herd guard (bd-gpeot, epic bd-2cenq).** New `backend/services/emby_cache.py` wraps `EmbyClient.get_sessions()` so the BandwidthTracker poll loop (~5s cadence, bd-gih6d follow-up) and the resolver (bd-6802c follow-up) can read the live Emby session list without pounding the operator's Emby server on every cycle. **Module-level `async get_cached_emby_sessions() -> list[EmbySession]`** is the only public entry point; cache key is the constant `"emby:sessions"` (single-key — Emby is one configured upstream per ECM install) and TTL is 5 seconds (matched to the Dispatcharr poll cadence so the next BandwidthTracker poll sees a fresh snapshot and intermediate readers hit cache). **Settings gate** short-circuits before any cache or client interaction when Emby is disabled or unconfigured: `getattr(settings, "emby_enabled", False)` is False OR `getattr(settings, "emby_base_url", "")` is empty → return `[]` immediately, no `EmbyClient` constructed. The `getattr` defaults defensively handle the pre-bd-8wc6q state where the `DispatcharrSettings` model has not yet declared the `emby_*` fields; once bd-8wc6q lands, the defaults become inert. **Stale-fallback** is the deliberate departure from the generic `cache.py` helper: when the upstream fetch fails AFTER a value was previously cached, the stale list is returned (with a `[EMBY]` WARN naming the failure and the stale age) rather than empty list — a momentary Emby outage should not erase the "who is watching" picture mid-resolver. Cold-start failure (no cache + fetch fails) still returns `[]` (always a list, never `None` to callers) and logs a `[EMBY]` WARN with `"no prior cache"` phrasing so operators can distinguish the two failure modes in logs. Cold-start failure deliberately does NOT poison the cache with `[]` — the next call retries against Emby rather than serving empty for the TTL window. **Thundering-herd prevention** via a lazy-initialized module-level `asyncio.Lock` (constructed on first call so module import is not bound to a running loop): concurrent callers during a cache-miss queue at the lock; the holder fetches once and the waiters re-check the cache under the lock (double-checked locking — without the re-check, every waiter would still fire its own fetch after acquiring the lock, defeating the guard entirely) and return the freshly-cached value. The `EmbyClient` is constructed and closed per fetch (httpx connection pool released on success AND failure via try/finally so a transport error never leaks sockets). Logging uses the `[EMBY]` prefix with lazy `%` formatting per `backend/CLAUDE.md`. Caching is deliberately the only concern here: resolver / matching logic is bd-6802c's domain (next wave), BandwidthTracker wiring is bd-gih6d's domain (next wave), settings fields are bd-8wc6q's domain (parallel). **Tests:** 11 new tests in `backend/tests/services/test_emby_cache.py` lock the bead's full behavioral matrix — (1) cache hit: first call constructs one `EmbyClient` and fires one `get_sessions`, client closed once; (2) cache hit: second call within TTL returns cached value without constructing a new client; (3) TTL expiry: monkey-patched `time.monotonic` past the TTL boundary triggers a refetch and the new value wins; (4) stale-fallback: prior success + later `EmbyClientError` returns the stale list with a WARN containing `"stale"`; (5) cold-start failure: no prior cache + `EmbyClientError` returns `[]` with a WARN containing `"no prior cache"`; (6) cold-start failure does not poison the cache: subsequent successful fetch fires a new upstream call rather than serving an empty cached value; (7) disabled (`emby_enabled=False`) returns `[]` without constructing a client; (8) empty `emby_base_url` treated as disabled (defensive against the bd-8wc6q-not-yet-shipped state); (9) settings model without any `emby_*` attributes treated as disabled (bare-object stand-in for pre-bd-8wc6q `DispatcharrSettings`); (10) disabled short-circuit even with a pre-populated cache (operator disabling Emby after a cache load must NOT surface stale data); (11) thundering-herd: 10 concurrent `asyncio.gather` callers during cold-start fire exactly ONE `EmbyClient` construction and ONE `get_sessions()` call, all 10 receive the same session list — implemented with a real `CountingClient` counter class (not Mock) for deterministic concurrent counting and an `asyncio.Event` gate to hold the in-flight fetch open until all waiters arrive at the lock. - **Emby user attribution columns on `session_telemetry` (bd-k026g, parent epic bd-2cenq).** Schema substrate for the Emby user attribution epic — ECM only sees the Dispatcharr stream session's IP, and when users watch via an Emby server ALL stream pulls share that one IP, so Stats cannot distinguish individual Emby viewers without enrichment. **Alembic migration 0016** adds two NULLABLE TEXT columns to `session_telemetry`: `emby_user_id` (TEXT NULL — Emby user IDs are GUIDs, NOT integers like the Dispatcharr `user_id` column) and `emby_user_name` (TEXT NULL — denormalized at write time from the per-poll Emby `/Sessions` lookup the resolver maintains, mirroring the bd-gsn3r `dispatcharr_username` denormalization). Plain nullable columns, NOT FKs — Emby's user table is upstream and not an ECM table. **Schema posture is deliberately denormalized** per the bead spec: keeping Stats v2 query patterns flat (per-poll projection alongside `dispatcharr_username`) avoids the per-row JOIN cost a separate `emby_users` table would impose; the resolver lookup that populates these columns already runs once per poll window. **Idempotency (bd-5w6jz):** each ADD COLUMN is independently guarded via `inspect()` so partial drift (one column already present from `create_all()` against the post-0016 ORM, or from a previous interrupted run) is safe to re-encounter; if both columns are already present, the migration returns immediately. - **Emby API client + auth (bd-6c0g6, epic bd-2cenq — foundational module for the v0.17.x Emby user-attribution feature).** New read-only async HTTP client at `backend/emby_client.py` modeled on `dispatcharr_client.py` — single concern: fetch the operator's Emby `/Sessions` feed so downstream code (bd-gpeot TTL cache, bd-6802c resolver, BandwidthTracker enrichment) can cross-reference live Emby viewers against ECM's active streams and attribute the real Emby username instead of collapsing every Emby-mediated pull to the proxy IP. **`EmbyClient(base_url, api_key)`** normalizes the base URL (strips a trailing slash but preserves any reverse-proxy sub-path) and constructs an `httpx.AsyncClient` with the bead-mandated timeout (`httpx.Timeout(connect=5.0, read=10.0)`); the auth scheme is stateless `X-Emby-Token: <api_key>` per request, no token-refresh lifecycle (unlike Dispatcharr's JWT flow). **`async get_sessions() -> list[EmbySession]`** calls `GET <base>/Sessions` and maps the upstream PascalCase payload to the snake_case `EmbySession` frozen dataclass (`session_id` ← `Id`, `user_id` ← `UserId`, `user_name` ← `UserName`, `remote_endpoint` ← `RemoteEndPoint`, `now_playing_item_name` ← `NowPlayingItem.Name`, `now_playing_channel_name` ← `NowPlayingItem.ChannelName`, `last_activity_date` ← `LastActivityDate`); idle sessions (no `NowPlayingItem`) produce `None` for the playing fields rather than raising; empty `/Sessions` responses return `[]` (a normal idle-server state, not an error). **`EmbyClientError`** wraps every failure path — 401 (bad/expired API key) surfaces a distinct message string the bd-8wc6q Settings UI can route on; any other non-2xx response surfaces the status code; transport-level failures (`httpx.ConnectError`, `ReadTimeout`, etc.) wrap the underlying exception in the `__cause__` chain so structured loggers can still capture root cause. **`async test_connection() -> bool`** wraps `get_sessions()` for the bd-8wc6q Settings UI 'Test Connection' button — returns `True` on any 2xx response, swallows `EmbyClientError` to `False` so the UI handler only needs to render a bool. - Auto-creation now strips letter-superscripts (ᴿᴬᵂ → RAW, ᴴᴰ → HD) from channel names it creates, while still preserving numeric superscripts (ESPN² stays ESPN²). PR #61 added a `preserve_superscripts=True` flag to `normalize()` that short-circuited ALL superscript conversion — including letter-superscripts used as quality-tag cruft — so auto-creation output retained `ᴿᴬᵂ`/`ᴴᴰ` even when Settings → Normalization's "Test" button stripped them. Fix splits the internal superscript map into `LETTER_SUPERSCRIPTS` (always converted) and `NUMERIC_SUPERSCRIPTS` (preserved when flag set); `convert_superscripts()` gains a `preserve_numeric` argument; the `preserve_superscripts` call-site in `normalize()` now routes through the letter-only path. Backward-compatible: the public `SUPERSCRIPT_MAP` still exists as the union of both (bd-yui1k, supersedes coarser PR #61 behavior).
- Stream probing now works on HTTPS streams. The ffprobe/ffmpeg `-protocol_whitelist` was missing `tls` (and `crypto`), so every HTTPS `.ts`/`.m3u8` source failed with `[tls @ ...] Protocol 'tls' not on whitelist ...!` and `Invalid argument`. Added `tls,crypto` to the whitelist in `backend/stream_prober.py`, `backend/ffmpeg_builder/probe.py`, and `backend/routers/stream_preview.py`. Neither is a user-supplied URL scheme (they are internal demuxers chained from `https://` / encrypted HLS), so the existing threat model against `file://`, `data:`, `concat:`, `subfile:`, etc. is preserved. New regression tests lock `tls` and `crypto` into the whitelist and assert dangerous schemes stay out (GH-106, bd-4awsl).
- Auto-creation rule duplicate preserves stream/channel sort, normalization groups, orphan cleanup, and option toggles — duplicates now match the original's configuration instead of silently dropping fields (PR #94, external contributor stevencoutts, bd-x3lto).
- Auto-creation no longer creates duplicate channels when an existing channel's name differs only by a normalization-matched suffix (e.g., "RTL ᴿᴬᵂ" and incoming "RTL"). Root causes: normalization engine wasn't initialized unless at least one rule opted in via normalize_names=True, and the channel-lookup map only indexed stored-name → normalized. Fix: always initialize engine when any NormalizationRuleGroup is enabled; add symmetric normalized-search + core-name fallbacks in _find_channel_by_name (PR #107, bd-u9odj, GH-104).
- Frontend test `NormalizationEngineSection.applyToChannels.test.tsx` no longer hangs in CI — the `useNotifications` test mock returned a fresh object on every call, which invalidated the component's `loadData` `useCallback` and triggered an infinite re-render loop that kept the test stuck in the loading state. Mock now returns a stable singleton (PR #107, bd-u9odj).
- Auto-Creation Pass 3.5 stream reorder now runs on channels modified via `merge_stream`/`merge_streams_prune` — previously silent when Channels Created = 0, leaving streams unsorted after a pure-merge run. (PR #113, enhancedchannelmanager-f55hm, gh-113)
- Auto-Creation name-based stream sort now works when probe stats are unavailable — the reorder path now seeds stream names from the channel's stream payload so natural-order and alphabetic sort have something to sort on. (PR #113, enhancedchannelmanager-f55hm, gh-113)
- Auto-Creation quality (resolution) sort now honors `deprioritize_failed_streams` — black-screen / failed / low-fps streams no longer promote to the top ahead of working higher-resolution streams. (PR #113, enhancedchannelmanager-f55hm, gh-113)
- `journal_entries` table now has the indexes its access patterns require. bd-91mcq introduced per-entity journal rows with a shared `batch_id` for bulk auto-creation operations, but the table only had indexes on `timestamp DESC`, `category`, and `action_type` — so the forensic "show me everything in batch X" lookup full-scanned, and "history for entity N" queries fell back on the timestamp index. Adds Alembic revision `0004` creating `idx_journal_batch_id` on `(batch_id)` and `idx_journal_entity` on `(category, entity_id, timestamp DESC)`; mirrors them in the SQLAlchemy model so model + migration stay in sync (drift test confirms). Also raises `CleanupTask.journal_days` default from `30` → `90` to align with `journal.purge_old_entries`'s existing 90-day default — bulk operations now amplify journal row volume N-fold, and the inconsistent retention default would have masked unbounded table growth. CleanupTask was already wired into the scheduler; no integration changes required (PR #133, bd-dmu8w).
- Stats tab no longer crashes with React error #185 (Maximum update depth exceeded) on every load. Recharts 3.x bundles `react-redux` + `@reduxjs/toolkit` for internal state; under React 19's stricter concurrent rendering, the chart's `YAxis` → `RenderedTicksReporter` `useEffect` re-dispatched `setRenderedTicks` with a fresh `ticks` array reference on every render, which forced a re-render that re-ran the effect, looping until React aborted with #185 inside the ErrorBoundary scoped to the Stats tab. The trigger was the React 18.3.1 → 19.2.5 bump in `ca6e5019` (bd-in620, 2026-04-24); the recharts 3.7.0 → 3.8.1 bump in `62ea5c5f` (2026-04-16) seeded `RenderedTicksReporter` itself, but the loop only manifested under React 19. Pin `recharts` to `2.15.4` — the latest 2.x line, declares React 19 support in peerDeps, and pre-dates the redux refactor entirely. Our chart usage (`LineChart`/`AreaChart`/`BarChart` + `XAxis`/`YAxis` + `Tooltip` + `ReferenceLine` + `ResponsiveContainer`) is all stable across the 2.x→3.x boundary; tsc clean and 110 stats panel tests pass against 2.15.4 (bd-a7l1z).

### Security
- `backend/routers/backup.py` — canonicalize-and-verify (`resolve()` + `relative_to()`) applied to `download_saved_backup` and `delete_saved_backup` as defense-in-depth alongside the existing `_BACKUP_FILENAME_RE` guard. Closes CodeQL py/path-injection alerts 1416-1419 (CWE-22/23/36/73/99). Pattern mirrors the zip-restore remediation in the same file. New tests cover traversal, absolute paths, symlink-escape, and null bytes (PR #98, bd-0a1pr).
- `backend/cloud_storage/onedrive_adapter.py` — Pydantic + adapter-level validators on `tenant_id` (GUID or verified domain) and `drive_id` (base64url-style) prevent SSRF via Microsoft Graph URL interpolation. Closes CodeQL CRITICAL py/partial-ssrf alerts 1361, 1362 (CWE-918). New tests at `backend/tests/test_cloud_storage.py` cover validator unit coverage + HTTP 422 at the cloud-target API (PR #101, bd-zbt74).
- `backend/auto_creation_schema.py` — three write-time `re.compile(<user_input>)` sites (condition regex validation, set_variable pattern validation, name_transform_pattern validation) now route through `safe_regex.compile`, which enforces a length cap and uniform compile-error surface. User-controlled regex from auto-creation rule definitions can no longer pin a worker via crafted ReDoS patterns. The bd-91mcq auto-creation bulk merges expanded request volume through these paths; this hardening closes the exposure window. Removes the three `# nosemgrep: no-bare-re-on-dynamic-pattern` annotations + their `TODO(enhancedchannelmanager-ltjyx)` markers. Backward-compatible: error message substrings (`"Invalid regex"`, `"Invalid name_transform_pattern"`) preserved for existing test/UI assertions. New `TestSafeRegexMigrationWriteTimeValidation` (10 tests) covers oversize-pattern rejection at all three sites + boundary-at-cap acceptance (PR #136, bd-ltjyx).
- `backend/routers/m3u_digest.py` — two write-time `re.compile(<user_input>)` sites validating `exclude_group_patterns` and `exclude_stream_patterns` from request bodies now route through `safe_regex.compile`. Same ReDoS class as bd-ltjyx, different surface (M3U digest endpoint instead of auto-creation rule endpoint). Removes the two `# nosemgrep` annotations. New tests in `TestUpdateDigestSettings` cover oversize-pattern rejection on both fields plus a `TestRouterUsesSafeRegex` source-level revert guard (PR #135, bd-3u6p0).
- Eight CodeQL py/stack-trace-exposure HIGH alerts (1412-1415, 1350-1353) remediated across `backend/routers/{backup,channels,export,health,settings}.py` + `backend/cloud_storage/`. Pattern: `logger.exception(...)` for full trace correlated by `X-Request-ID` + return only the exception class name (`type(e).__name__`) in the API response — never `str(e)` (which can include connection strings, file paths, or credentials). For `ImportError` paths in cloud_storage adapters, `e.name` (the missing module name) is used so the response stays useful without leaking interpreter paths. **Material risk closed**: alert 1414 in `health.py` — `/api/health/ready` is in `AUTH_EXEMPT_PATHS`, meaning DBAPI exception messages were echoed unauthenticated; this PR closes that. ADR-005 Phase 1 disallows "won't fix" dismissal so every open stack-trace alert needed a real fix. Six new sanitization tests + 1 updated, each asserting both the positive contract (response equals exception class) and the negative contract (the secret-bearing message body MUST NOT appear in the response) (PR #160, bd-m8i9q).
- `backend/export_manager.py` + `backend/routers/export.py` — defense-in-depth canonicalize-and-verify backstop for path traversal in the M3U/XMLTV export download paths. New `_safe_export_path(profile_id, *parts)` helper resolves and verifies containment under `EXPORTS_DIR.resolve()` via `Path.relative_to()`, raising `ExportPathError` on escape. Wired into `ExportManager.get_export_path` and `ExportManager.cleanup` (cleanup short-circuits before `rmtree` on escape). Also re-verifies path containment in `download_m3u`/`download_xmltv` before serving. CodeQL py/path-injection alerts 1354-1359 were already dismissed as false-positive (FastAPI int path-param + Pydantic `FILENAME_RE` allowlist make traversal unreachable upstream), but per ADR-005 sequencing 2 and the bd-0a1pr precedent the property is now guaranteed at the sink rather than depending on caller hygiene. New `TestExportPathInjection` (9 tests) mirrors `TestSavedBackupsPathInjection` from bd-0a1pr — traversal payloads, absolute paths, null-byte, backslash, cleanup sentinel-file regression, symlink-escape (PR #159, bd-h5rfv).

## [0.15.1] — 2026-03-31

### Added
- Login rate limiting — 5 attempts per minute per IP via slowapi on both local and Dispatcharr login endpoints.
- Security headers: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`.
- Client IP in login-failure logs (includes `X-Forwarded-For` for forensics).

### Changed
- **Password policy now follows NIST 800-63B** — dropped composition rules (uppercase/lowercase/number), replaced the 33-word list with the 10k common-passwords list, uniform 8-character minimum for all users including admin-created.
- CORS tightened — explicit method and header allow-lists instead of wildcards.

### Fixed
- EPG matching: channel names with a number prefix (e.g. `535 | ESPN`) were normalizing to `535espn` instead of `espn`, causing zero matches against EPG data.

### Security
- Injection / SSRF (OWASP A03/A10):
  - ffmpeg protocol allow-list on all invocations — blocks `file://`, `data://`, `concat:` and other dangerous protocols.
  - URL-scheme validation at input time on M3U, Xtream Codes, and EPG source endpoints — only `http://` and `https://` allowed.
  - Zip backup path canonicalization — defense-in-depth `resolve()` check alongside existing traversal guards.
- Logging hardening (OWASP A09):
  - 500-response scrubbing — global handler returns generic `Internal server error` to clients; real error stays in server logs.
  - Validation-error log redaction — `Authorization` / `Cookie` headers and auth-endpoint request bodies redacted from logs.

## [0.15.0] — 2026-03-29

### Added
- **Export & Publish** — generate M3U/XMLTV playlists from channel profiles and publish to cloud targets (S3, WebDAV, etc.) on a cron schedule. Includes channel selector, playlist preview, cloud-target management, and publish history.
- **Low-FPS detection** — streams below a configurable threshold (5/10/15/20 FPS) are flagged with an amber icon and deprioritized in Smart Sort. Zero-overhead, always-on, configurable in Settings → Maintenance.
- **Black-screen detection** — optional ffmpeg `signalstats` check after each probe flags dark/blank streams with a purple icon. Configurable sample duration (3–30s) in Settings → Maintenance.
- **Normalize Names engine** — tag-based rules engine for cleaning up stream names during bulk channel creation. YAML import/export, edit/revert controls, drag-and-drop priority ordering.
- **Backup & Restore** — full configuration backup/restore from Settings, with restore option during the first-run setup wizard.
- **Merge Channels** — combine two or more channels into one, consolidating their streams.
- Auto-creation enhancements — EPG logo-source action, Probe Streams action, Set Channel Profile action; Smart Sort option in rule sort dropdown; rule selection in Auto-Create schedule; separate stream sort from channel sort per rule.
- PUID/PGID support for Docker container user identity.
- TV Guide print view.
- JWT authentication in Swagger UI with `/swagger` redirect.
- Debug bundle for troubleshooting auth issues.

### Changed
- Server-side migration of EPG matching, stream normalization, and edit consolidation (previously client-side).
- Settings UI consistency — compact dropdowns, unified admin section CSS, proper field ordering.
- Scoped reprobe to last scheduled probe's channel groups.
- Capped Dispatcharr HTTP client connection pool.
- All GitHub Actions updated to Node.js 24-compatible versions.
- Migrated 161 `console.log` call sites to the structured logger.
