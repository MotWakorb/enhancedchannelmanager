- **Bulk M3U import dedup hook populates the `pending_merges` queue (bd-a5lb2 / BD-F).** When the auto-creation pipeline runs after an M3U refresh and would create a new channel from an incoming stream, the new bulk-M3U dedup hook (`backend/services/m3u_dedup_hook.py`) first scores the stream name against existing channels in the same target group using BD-A's matcher (`services.dedup_matcher.find_candidate`). The hook fires AFTER the executor's existing exact / normalized lookups (`_find_channel_by_name`) have failed and BEFORE the new-channel `create_channel` API call, so all current "merge into existing" and "skip if exists" paths are unchanged. **ADR-008 ┬¦D5 partial-unique-index idempotency:** repeat M3U refreshes of the same stream against the same candidate produce one INSERT then catch `IntegrityError` on subsequent attempts, logging at INFO (`[DEDUP] Pending merge for stream=... candidate=... already queued; skipping`) and returning "skip channel creation" because the prior pending row is still authoritative ŌĆö no crash, no duplicate row. **ADR-008 ┬¦D9 broker-free:** writes go directly to `pending_merges` via the existing ECM session; no Celery, no Redis, no scheduler daemon. **Hook is gated to `triggered_by='m3u_refresh'`:** scheduled / manual auto-creation runs predate the dedup epic and stay on legacy "always create" semantics ŌĆö they are not one of the four ADR-008 ┬¦D1 interactive trigger surfaces. **Dry-run preview** never mutates the queue. **LOCKED CONTRACT metric (BD-M):** new `Counter` `ecm_pending_merges_queue_depth_added_total` registered in `backend/observability.py`, +1 per fresh INSERT (collisions do NOT increment ŌĆö they are not new rows). ### Fixed
- **Auto-creation Smart Sort: custom streams (operator-added, non-M3U) are now included in priority ordering (bd-sgtmx, GH #244).** Previously the Smart Sort logic silently excluded custom streams, leading to confused operator-created channel definitions for groups containing both M3U and custom streams. Fix preserves all existing tie-break behavior; only the inclusion logic changes. Custom streams (those with no backing M3U account) now participate in the `m3u_priority` sort criterion via a reserved `"custom"` key in `Settings ŌåÆ M3U account priorities`. Example: `{"1": 100, "custom": 200}` places all operator-added streams above M3U account 1. When `"custom"` is not set the default is `0`, matching the pre-fix treatment ŌĆö no behavioral change for operators who leave it unset. The fix applies to both the auto-creation engine (Pass 3.5 stream reorder) and the stream-prober smart sort so behavior is consistent across both surfaces. ([bd-sgtmx], [GH #244])
- **`settings.json` `api_key` field-name collision with the MCP integration (bd-jmi1c, GH #273).** Resolves GH #273 by renaming the Dispatcharr REST API token field from `api_key` to `dispatcharr_api_key`, with one release of back-compat for the legacy name. Three distinct credentials shared a confusingly-overloaded namespace in `settings.json`: ECM read `api_key` as its Dispatcharr REST API token, the MCP "Generate API key" flow correctly writes `mcp_api_key`, and the Settings UI labels both as "API Key". Operators following the redeployment procedure (Settings ŌåÆ MCP Integration ŌåÆ Regenerate Key, copy the new key) were placing the MCP key in `api_key`, which made ECM send the MCP key to Dispatcharr and return 401 on every channel/stream operation. The v0.17.0 `bd-ix1g6` `/health` self-diagnosis helped operators recognize a missing MCP key but did not address this collision. The rename eliminates the lexical hazard at the source. **Backend (`backend/config.py`):** new canonical `dispatcharr_api_key` field on `DispatcharrSettings`; legacy `api_key` field retained as a deprecated alias for v0.17.x with a one-time-per-process `[CONFIG] Reading deprecated 'api_key' field as Dispatcharr token ŌĆö please rename to 'dispatcharr_api_key' in settings.json` WARN on first read; `load_settings()` silently migrates legacy ŌåÆ canonical in the loaded dict (canonical wins when both are populated); `save_settings()` mirrors canonical ŌåÆ legacy on write so external scripts that read `settings.json:api_key` directly (the workaround in the GH #273 issue body) keep working until the legacy field is removed in a future release. **Backend (`backend/dispatcharr_client.py`):** the X-API-Key header reads `dispatcharr_api_key` with a legacy fallback; the `_settings_hash` includes both fields so the singleton client recreates on any rotation. **Backend (`backend/routers/settings.py`):** `SettingsRequest` accepts either field name (canonical wins when both are present); `SettingsResponse` surfaces both `dispatcharr_api_key_configured` (canonical) and `api_key_configured` (legacy alias) so older cached frontend bundles keep working; `TestConnectionRequest` accepts either field name. **Backend (`backend/routers/backup.py`):** `_SETTINGS_CREDENTIAL_FIELDS` lists both `dispatcharr_api_key` and `api_key` so the backup redactor never leaks either form. **Frontend (`frontend/src/services/api.ts`, `SettingsModal.tsx`, `SettingsTab.tsx`):** type definitions and POST bodies switched to the canonical field; the indicator-state hook prefers `dispatcharr_api_key_configured` with a fallback so a new frontend bundle still works against an older backend mid-rollout. **Tests:** 12 new unit tests in `backend/tests/unit/test_config_dispatcharr_api_key_migration.py` cover the canonical-only read, legacy-only read with WARN, both-populated canonical-wins, one-WARN-per-process discipline, save-time canonicalŌåÆlegacy mirror, and full loadŌåÆsaveŌåÆload idempotency. 4 new integration tests in `backend/tests/routers/test_settings.py` verify the POST endpoint accepts both field names, picks canonical when both are sent, and GET returns both indicators. **Operator migration path:** existing v0.17.x operators see one `[CONFIG] Reading deprecated 'api_key' field ŌĆ”` WARN on the first start after upgrade; the next save (any settings change, or the next MCP key rotation) silently mirrors the value into the new canonical field. No manual `settings.json` edit is required. Manual cleanup (removing the legacy `api_key` line) is allowed once the operator has confirmed `dispatcharr_api_key` is populated. **Field-rename scope is intentionally narrow:** `mcp_api_key` is correctly named and unchanged; the `auth_method: "api_key"` enum value is unchanged (it is the mode name, not the field name); the bd-ix1g6 `/health` MCP self-diagnostic is unchanged. Legacy 'api_key' field is deprecated and will be removed in v0.19.0 (see bd-ewm4h). ([bd-jmi1c], [GH #273])
- **Bulk channel merge: pre-validate source IDs to return 422 instead of silently producing DELETE 404 noise on stale rows (bd-ozhkf ŌĆö follow-up to bd-ct9wl).** The `bulk_merge_channels` endpoint had the same bare-`except` footgun as the single-channel `merge_channels` endpoint before bd-ct9wl: a `get_channel` 404 was swallowed and execution fell through to `delete_channel`, generating `[DISPATCHARR] API request failed: DELETE 404` log noise for every stale source ID. The fix mirrors the bd-ct9wl pattern exactly: catch `httpx.HTTPStatusError` with `status_code == 404`, collect missing IDs, raise HTTP 422 with the same detail string (`"Source channels [...] no longer exist ŌĆö refresh the channels list and try again"`) before any mutations. Two regression tests in `TestBulkMergeChannelsStaleSourceIds` assert the 422 path and that `delete_channel` is not called on stale IDs.
- **Channel merge UX: surface backend 422 detail to operator instead of swallowing into generic "Merge failed" error (bd-7j6v1 ŌĆö follow-up to bd-ct9wl).** The `httpClient.ts` `fetchJson` function already extracts the FastAPI `detail` field from non-2xx responses and throws it as `HttpError.message`, so `MergeChannelsModal` and `FindDuplicatesModal` already propagate the 422 detail string to the error banner through their `setError` catch handlers ŌĆö no code change required. Added regression tests to lock this contract: `MergeChannelsModal.test.tsx` and `FindDuplicatesModal.test.tsx` each assert 422 detail renders verbatim in the error banner, a non-422 `HttpError` renders its message, and a non-`Error` thrown value falls back to the generic copy.

### Fixed
- **Ghost source channels after "Merge channels" in edit mode (bd-ct9wl).** With edit mode active, selecting 2+ channels and choosing "Merge channels" merged them server-side (target channel got the streams, sources were deleted) but the source rows remained visible in the channels grid. The cause was the `useEditMode` working-copy sync effect only adding new channels ŌĆö it never removed channels that disappeared from the API. The downstream consequence was operator-visible 404 noise: a user who then re-selected ghost rows for a second merge sent the now-stale source IDs to the backend, which forwarded the DELETEs to Dispatcharr and got `404 Not Found` per source (logged as `[DISPATCHARR] API request failed`), leaving the operator with no useful error to act on. The sync effect now also removes channels with persisted IDs that vanish from the API, while preserving locally-created not-yet-saved channels (negative temp IDs from `stageCreateChannel`) so an in-progress local-add does not get nuked by a routine API refresh. Two paired regression tests in `frontend/src/hooks/useEditMode.test.ts` lock the fix and the persisted-ID guard. Defense in depth: `/api/channels/merge` now catches a 404 from `get_channel(source_id)` and returns 422 with `"Source channels [...] no longer exist ŌĆö refresh the channels list and try again"` instead of falling through to a generic 500. Backend regression test in `test_channels.py::TestMergeChannelsStaleSourceIds`.
