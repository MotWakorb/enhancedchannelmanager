# Changelog

All notable changes to Enhanced Channel Manager are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- Stats v2 Active Channels live stream-identity badge: `GET /api/stats/channels` now enriches each active-channel row with `stream_name` and `m3u_account_id` resolved at request time via the same resolver chain the BandwidthTracker poll uses (direct `stream_id` → URL-derived → channel-streams cross-ref). UI: StatsTab Active Channels section renders a `[<provider>] - <stream_name>` badge next to the channel name (e.g., `[Infinity] - US: TNT`) using `streamLabel` helper from `bd-kh23e` and the `getM3UAccounts` side-load pattern from `bd-vjv7k`. Edge cases: when `m3u_account_id` is unmapped or NULL, falls back to bare `<stream_name>` (no `[Provider <id>]` leak); when `stream_name` equals the channel display name, suppresses the badge to avoid duplication; when both are absent, no badge. Shared resolver entry point `resolve_active_channel_streams()` extracted to `bandwidth_tracker.py` as a free function; the endpoint instantiates a fresh `ChannelStreamsCache` per request and passes `emit_metrics=False` so on-demand calls don't dilute the cyclic-poll SLI signal owned by the polling hot path. Closes the "I don't see the stream name next to the active channels" gap (`bd-kh23e` covered historical aggregates only, not the live view). ([bd-ox5q8])
- Stats v2 stream-identity capture: `session_telemetry` persists `stream_id` and `stream_name` per row (Alembic `0010`). Resolver refactored to return a `ProviderResolution(provider_id, stream_id, stream_name)` NamedTuple per channel. APIs `GET /api/stats/watch-time/{user_id}` and `GET /api/stats/providers/channel-heatmap` gain `latest_stream_id` + `latest_stream_name` (picked via `MAX(observed_at)` per channel / per `(provider, channel)`). UI: Users panel per-user breakdown adds a "Stream" column; Providers panel heatmap data-table shows `[<provider>] - <stream>` (e.g., `[Infinity] - US: TNT`). Format helper at `frontend/src/utils/formatting.ts` falls back to bare stream name when provider context isn't available. Closes the "we don't show which stream is playing" gap. ([bd-kh23e])

### Fixed
- Stats v2 User Watch Time panel no longer attributes Dispatcharr-side viewer rows to the ECM admin/API account. ECM's local `users` table and Dispatcharr's `users` table are separate namespaces with coincidentally-overlapping integer ids — when ECM `users.id=3` is the local admin ("claude") and Dispatcharr `users.id=3` is a real human viewer, the Stats panel renders the human's viewing sessions under the admin's name (the panel joins `session_telemetry.user_id` against ECM's `users` table). New env var `ECM_TELEMETRY_EXCLUDE_USERS` (comma-separated, default empty/unset = no filtering, no behavior change) suppresses `_write_session_telemetry` rows whose Dispatcharr-side `user_id` OR resolved username matches any token (case-insensitive). New Prometheus counter `ecm_session_telemetry_rows_excluded_total{reason="excluded_user"}` surfaces suppression rate honestly — skipped writes are counted, not silently zeroed. Per-row `[BANDWIDTH] Skipped telemetry write for excluded user user=… reason=excluded_user` debug log for operator traceability. Backfill of existing mis-attributed rows is operator-driven (NOT auto-applied at startup): `docker exec ecm-ecm-1 python3 -c "import sqlite3; c=sqlite3.connect('/config/journal.db'); n=c.execute('DELETE FROM session_telemetry WHERE user_id=?', (3,)).rowcount; c.commit(); print(f'deleted {n} rows')"` (substitute the local admin's `users.id`). ([bd-uqbob])
- Stats v2 Providers panel polish — "Time spent per provider" + "Bitrate by provider" charts are now self-documenting. The time-spent chart was a single-bucket stacked `AreaChart` rendering one tall "Total" stack with no Y-axis label, so a PO couldn't tell from the chart which provider was which or what the Y-axis values meant. It's now a per-provider `BarChart`: one bar per provider with the provider name on the X-axis, watch minutes on the Y-axis, an explicit "Watch minutes" axis label, and a one-line description above the chart. The bitrate chart already rendered per-provider time-series correctly with auto-scaled tick formatting (bps / Kbps / Mbps), but had no Y-axis label and no caption explaining what the metric was; both are now in place — Y-axis labeled "Bitrate (auto-scaled)" and a one-line description noting the formula derivation (`SUM(bytes_delta) * 8 * 1000 / SUM(poll_interval_ms)` per `(provider, time_bucket)`, equivalently `bytes / seconds * 8`). Backend bitrate + watch-time endpoints are unchanged — both already return per-provider rows and the bitrate formula was already correct; two new contract-shape regression tests (`test_response_row_shape_matches_documented_contract`) pin the field names the panel binds to. ([bd-tknci], [bd-zrk05])
- Stats v2 provider resolver: channel-streams URL-match fallback for the upstream-id case. kbgey's URL fallback parses the trailing `<id>.ts` integer from the active stream URL, but empirically that integer is the **upstream M3U provider's** stream id (e.g. Infinity's `85796`), NOT Dispatcharr's stream row id — `get_streams_by_ids([85796])` returns nothing, the resolver previously gave up with `reason=stream_not_found_url_derived`, and `provider_id` stayed NULL (Unknown bucket). The resolver now falls back to `GET /channels/<channel_id>/streams`, matches by URL substring, and uses the matched stream's `m3u_account_id`. Per-poll cache + small cross-poll LRU keeps the API call rate bounded. ([bd-5g7kx])
- Stats v2 session_telemetry: anonymous `user_id=0` (and `"0"`) no longer poison the per-poll batch commit. Dispatcharr returns `user_id="0"` for anonymous viewing sessions; `session_telemetry.user_id` is INTEGER NULL with an FK to `users.id`. The 0 sentinel doesn't exist in `users` → `sqlite3.IntegrityError` at `session.commit()` rolled back the ENTIRE batch including correctly-resolved rows. New `_coerce_session_user_id(raw) -> Optional[int]` helper normalizes 0 / "0" / "" / None / negative ints / bools / floats / unparseable strings to None; positive ints / int-parseable strings pass through. Defense-in-depth: `_write_session_telemetry` now also catches `IntegrityError` at commit time and logs `[STATS_V2] session_telemetry batch FK violation observed_at=... rows_attempted=...` so any future FK trip-up doesn't silently zero-out a batch. ([bd-gbxmj])

### Fixed (prior)
- Stats v2 provider resolver — URL fallback when Dispatcharr's `/proxy/ts/status` omits `stream_id`. The skqln.14 resolver previously gave up with `reason=no_stream_id` for any active channel where Dispatcharr didn't surface `stream_id` directly, leaving `provider_id=NULL` (the "Unknown" bucket). In dev, this hit 91% of polls (214 of 235). The resolver now also extracts the stream id from the URL path's trailing `<id>.ts` segment (`r"/(\d+)\.ts(?:\?|$)"`) and feeds it into the same `get_streams_by_ids` batch the resolver already runs per poll — no additional API calls. URL-derived lookups that miss log `reason=stream_not_found_url_derived` so ops can differentiate from the direct-stream-id miss case. ([bd-kbgey])
- Stats v2 Users panel — daily watch-minutes chart now displays dates in the operator's **local timezone** and visually marks today as "in progress" (amber dot vs teal for completed days), with a caption explaining the UTC bucketing + 10s update cadence. Resolves the misleading appearance of watch-time "going down" across the UTC date boundary when yesterday-UTC is complete but today-UTC is still accumulating. Server-side aggregation stays UTC; the relabeling anchors each bucket at noon UTC then formats in the operator's tz to pick the most-overlapping local day. ([bd-1qxo9])

### Fixed (existing entry continues below)
- Stats v2 Providers panel polish: provider rows now display the actual M3U account name instead of `Provider <id>`. Frontend side-loads the existing `/api/providers` endpoint at panel mount, builds an id→name map, and uses it in chart legends, heatmap row labels, and data tables. Falls back to `Provider <id>` for unmapped IDs (e.g., during the fetch); NULL stays "Unknown". ([bd-vjv7k])
- Stats v2 Providers heatmap polish: column labels rotated -45° (anchored at column center) and default `cellSize` bumped 32px → 48px; column-label band 24px → 80px to fit rotated text. Resolves the "everything super compressed" overlap when real channel names ("Discovery Channel", "ESPN HD") are wider than the column. The `.heatmap` container's existing `overflow-x: auto` handles wider heatmaps via horizontal scroll. Heatmap primitive interface preserved — `cellSize` prop still overridable for the time-of-day use case. ([bd-yteek])

### Added
- Stats v2 rollup tables + nightly prune (ADR-007 implementation): Alembic migration `0009` adds 3 new tables — `session_telemetry_user_daily` (PK `(user_id, channel_id, day)`) for the GH-62 Users panel rollup, `session_telemetry_provider_daily` (PK `(provider_id, channel_id, day)`) for the GH-59 Providers panel rollup, and `telemetry_rollup_state` (marker table per rollup_name with last_completed_day/run_status/run_error). Nightly scheduled task `stats_v2_rollup` (registered via `task_registry`) aggregates the previous day's raw `session_telemetry` rows into both rollup tables (idempotent via `INSERT OR REPLACE`), updates the state marker, and **after successful rollup** prunes raw rows older than 30 days (per ADR-007 retention) — prune is **skipped on rollup failure** so raw data isn't lost. NULL provider_id raw rows bucket into the `'unknown'` sentinel string in the provider rollup; NULL user_id raw rows are excluded from the user rollup (no behavioral subject to attribute, documented on the model). Watch-seconds use the DISTINCT-`(channel, observed_at)` collapse pattern; bytes_delta_sum and buffer_event_count are summed across raw rows since they're per-poll-per-client measurements. 5 new Prometheus metrics: `ecm_telemetry_rollup_last_success_timestamp{rollup_name}`, `ecm_telemetry_rollup_duration_seconds{rollup_name}`, `ecm_telemetry_rollup_days_processed{rollup_name}`, `ecm_telemetry_raw_rows_pruned`, `ecm_telemetry_rollup_errors_total{phase}` — all bounded-cardinality. ([bd-7i2vv])

Cross-poll dedup by `event.id` via a 10k-entry LRU (system-events feed re-delivers recent events; without cross-poll dedup the same event would be counted N times). Provider attribution piggybacks on skqln.14's `_resolve_provider_ids` — one shared resolver call per poll, not two. Count attributes to exactly **one row per (channel_uuid, observed_at) bucket** (the first emitted client-row gets the count, sibling rows write 0) so `SUM(buffer_event_count) GROUP BY provider, time_bucket` stays well-defined for the future per-provider read APIs without per-client double-counting. Events for channels that don't appear in the current poll's snapshot are dropped with `[STATS_V2] buffer_event_unmapped_channel` log. ([bd-skqln.15])
- Stats v2 provider attribution: `BandwidthTracker._collect_stats` now resolves the active stream's `m3u_account_id` for every channel in every poll and writes it as `session_telemetry.provider_id` (the data foundation the upcoming Providers panel and per-provider stats endpoints will read). Resolution piggybacks on the same `DispatcharrClient.get_streams_by_ids()` call ECM already uses for stream metadata — one batched API call per poll, not N. Resolver cache is scoped to a single poll cycle so a stream's failover hop is picked up next poll without staleness. Unresolvable channels (no active stream, mid-failover, or Dispatcharr returns the stream-without-provider) surface as `provider_id = NULL` and feed the "Unknown" bucket in the future provider panel — failures are logged with `[STATS_V2] provider_resolution_failed channel=... stream=... reason=...` and an SLI counter (`[STATS_V2] provider_resolution resolved=X unresolved=Y`) lands per-poll so `skqln.12` can wire it to Prometheus later. ([bd-skqln.14])

### Changed
- BandwidthTracker no longer writes to `ChannelWatchStats` — that aggregate table is now served by the `channel_watch_stats_v` view over `session_telemetry`. The legacy `ChannelWatchStats` table schema is retained for pre-cutover rows; no production code reads it post-skqln.3. Readers repointed in this release: `popularity_calculator._gather_metrics`, `BandwidthTracker.get_top_watched_channels` (the `/api/stats/top-watched` endpoint), and `BandwidthTracker._log_watch_stop_events`. `ChannelBandwidth` and `BandwidthDaily` writes are unchanged — they have independent live readers (per-day breakdown charts, bandwidth summaries). ([bd-skqln.3])
- Channel popularity formula's `watch_count` axis now uses `COUNT(DISTINCT session_id)` (distinct viewing sessions per channel) instead of the legacy state-transition counter. Operators may notice channel rankings shift visibly on the first calculator run after upgrade — a one-time event documented in `stats-v2-history-cutover.md`. The other three axes (`watch_time`, `viewers`, `bandwidth`) and the default weights are unchanged. ([bd-skqln.3])
- `ECM_SESSION_TELEMETRY_WRITE_ENABLED` environment-variable kill-switch retired. The flag's job ended when legacy writes were removed; `BandwidthTracker._write_session_telemetry` now runs unconditionally. Operators who set the env var manually can remove it from their compose / .env — it is silently ignored. ([bd-skqln.3])

## [0.16.0] — 2026-05-12

_This is the shipping 0.16.0. An earlier 0.16.0 build was cut on 2026-04-20 and rolled back before any consumer pulled it (see `[0.16.0-yanked]` below); everything from that attempt is included here, with the blocking bugs fixed._

### Added
- Auto-creation **Stream sort → Quality (resolution)** breaks ties at equal resolution using ECM **M3U account priorities** (same values as M3U Manager Save Priorities / Provider Order). Per-rule **`quality_tie_break_order`** (`desc` = higher priority first, default). Per-rule **`quality_m3u_tie_break_enabled`** (default on) disables that tie-break so equal-resolution ordering falls back to stream-id order only; Rule Builder and bulk-edit expose a checkbox and gate the direction dropdown. Two changes break a cross-component render loop that armed when the `ecm:open-task-editor` sessionStorage key was set (NotificationCenter writes it on Edit-Schedule clicks; ScheduledTasksSection only clears it after the matching task loads, so a stale or unmatched taskId stranded the key indefinitely): (1) the `useEffect` in `SettingsTab.tsx` that consumes the key was missing a dependency array and therefore re-ran every render — adding `[]` deps confines it to mount, breaking the loop driver; (2) `useHashRoute.setHash` always allocated a fresh `HashRoute` object literal and called `pushState` + `setRoute` even when the route was unchanged, so any caller invoking it with the same value (e.g., the buggy effect calling `setActivePage('scheduled-tasks')` with `activePage === 'scheduled-tasks'`) churned `history.length` and re-rendered every iteration — `setHash` now bails out when `tab` and `settingsPage` match the current route. Same family as the recharts Stats loop in `bd-a7l1z`; latent under React 18 batching, runaway under React 19.2.5 concurrent rendering. Browser verification with the trigger primed: 7 settings sub-page navigations produced exactly 7 history entries (pre-fix: tens of thousands within seconds), heap stable across idle and navigation (`bd-5xg17`, gh #207).
- Scheduled task email alerts now deliver via the SMTP Email alert method (recipients in `to_emails`) instead of being a no-op, and Notification Settings now exposes an "Email Alert Recipients" field to configure those recipients in the UI (`bd-bq27w`, `bd-lfmuw`).
- Scheduled task **success** emails (and shared alert metadata) no longer label **Auto-Create Channels** and **Stream Probe** with generic “Total Items” / “Success Count” that disagreed with the execution UI (e.g. tens of thousands “items” vs dozens of channel changes). `task_engine.py` now emits task-specific fields (streams evaluated/matched, channels created/updated, groups, conflicts, execution id / pipeline mode for auto-creation; streams scheduled/ok/failed/skipped and black-screen/low-FPS counts for probe) and human-readable success lines aligned with `TaskResult.message` / probe totals. `tasks/stream_probe.py` adds quality counters to `TaskResult.details`; `stream_prober.py` final external alert uses the same naming. Unit tests in `test_task_notification_formatting.py` (`enhancedchannelmanager-e3ts6`).
- Auto-creation `/run` and `/rules/{id}/run` no longer return `504 Gateway Timeout` on large catalogs. The bd-w3z4h request-timeout middleware (30s default) did not exempt `/api/auto-creation/`, so pipeline runs that exceeded 30s were cancelled client-side even though the work continued to completion server-side (the engine commits as it progresses). Added `/api/auto-creation/` to `_TIMEOUT_EXEMPT_PREFIXES` in `backend/main.py` alongside the other inherently long-running prefixes (`/api/tasks/`, `/api/backup/`, `/api/ffmpeg/`). Secondary symptom — the Execution History panel not auto-refreshing until browser reload — resolves with this fix because the frontend's post-run `fetchExecutions()` now runs on the success path instead of being skipped by the error. Follow-up refactor to convert `/run` to an async enqueue + polling pattern tracked as bd-enfsy (bd-zv6pi).
- Auto-creation no longer renumbers foreign-group channels that were matched via PR #107's normalized-name fallback. A rule with `sort_field` set would pick up any pre-existing channel found via the new normalized-search path — even channels owned by a different rule — and Pass 3 (channel renumber) would then reassign their channel numbers to fit its own sort. Fix: gate the Pass 3 append so a rule only tracks channels it created this run OR that were already in its pre-run managed set; cross-rule matches via fallback are still merged/skipped correctly but no longer trigger a foreign renumber (bd-yj5yi, GH-104 regression from PR #107).
- Auto-creation now strips letter-superscripts (ᴿᴬᵂ → RAW, ᴴᴰ → HD) from channel names it creates, while still preserving numeric superscripts (ESPN² stays ESPN²). PR #61 added a `preserve_superscripts=True` flag to `normalize()` that short-circuited ALL superscript conversion — including letter-superscripts used as quality-tag cruft — so auto-creation output retained `ᴿᴬᵂ`/`ᴴᴰ` even when Settings → Normalization's "Test" button stripped them. Fix splits the internal superscript map into `LETTER_SUPERSCRIPTS` (always converted) and `NUMERIC_SUPERSCRIPTS` (preserved when flag set); `convert_superscripts()` gains a `preserve_numeric` argument; the `preserve_superscripts` call-site in `normalize()` now routes through the letter-only path. Backward-compatible: the public `SUPERSCRIPT_MAP` still exists as the union of both (bd-yui1k, supersedes coarser PR #61 behavior).
- Stream probing now works on HTTPS streams. The ffprobe/ffmpeg `-protocol_whitelist` was missing `tls` (and `crypto`), so every HTTPS `.ts`/`.m3u8` source failed with `[tls @ ...] Protocol 'tls' not on whitelist ...!` and `Invalid argument`. Added `tls,crypto` to the whitelist in `backend/stream_prober.py`, `backend/ffmpeg_builder/probe.py`, and `backend/routers/stream_preview.py`. Neither is a user-supplied URL scheme (they are internal demuxers chained from `https://` / encrypted HLS), so the existing threat model against `file://`, `data:`, `concat:`, `subfile:`, etc. is preserved. New regression tests lock `tls` and `crypto` into the whitelist and assert dangerous schemes stay out (GH-106, bd-4awsl).
- Auto-creation rule duplicate preserves stream/channel sort, normalization groups, orphan cleanup, and option toggles — duplicates now match the original's configuration instead of silently dropping fields (PR #94, external contributor stevencoutts, bd-x3lto).
- Auto-creation no longer creates duplicate channels when an existing channel's name differs only by a normalization-matched suffix (e.g., "RTL ᴿᴬᵂ" and incoming "RTL"). Root causes: normalization engine wasn't initialized unless at least one rule opted in via normalize_names=True, and the channel-lookup map only indexed stored-name → normalized. Fix: always initialize engine when any NormalizationRuleGroup is enabled; add symmetric normalized-search + core-name fallbacks in _find_channel_by_name (PR #107, bd-u9odj, GH-104).
- Frontend test `NormalizationEngineSection.applyToChannels.test.tsx` no longer hangs in CI — the `useNotifications` test mock returned a fresh object on every call, which invalidated the component's `loadData` `useCallback` and triggered an infinite re-render loop that kept the test stuck in the loading state. Mock now returns a stable singleton (PR #107, bd-u9odj).
- Auto-Creation Pass 3.5 stream reorder now runs on channels modified via `merge_stream`/`merge_streams_prune` — previously silent when Channels Created = 0, leaving streams unsorted after a pure-merge run. (PR #113, enhancedchannelmanager-f55hm, gh-113)
- Auto-Creation name-based stream sort now works when probe stats are unavailable — the reorder path now seeds stream names from the channel's stream payload so natural-order and alphabetic sort have something to sort on. (PR #113, enhancedchannelmanager-f55hm, gh-113)
- Auto-Creation quality (resolution) sort now honors `deprioritize_failed_streams` — black-screen / failed / low-fps streams no longer promote to the top ahead of working higher-resolution streams. (PR #113, enhancedchannelmanager-f55hm, gh-113)
- `journal_entries` table now has the indexes its access patterns require. bd-91mcq introduced per-entity journal rows with a shared `batch_id` for bulk auto-creation operations, but the table only had indexes on `timestamp DESC`, `category`, and `action_type` — so the forensic "show me everything in batch X" lookup full-scanned, and "history for entity N" queries fell back on the timestamp index. Adds Alembic revision `0004` creating `idx_journal_batch_id` on `(batch_id)` and `idx_journal_entity` on `(category, entity_id, timestamp DESC)`; mirrors them in the SQLAlchemy model so model + migration stay in sync (drift test confirms). Also raises `CleanupTask.journal_days` default from `30` → `90` to align with `journal.purge_old_entries`'s existing 90-day default — bulk operations now amplify journal row volume N-fold, and the inconsistent retention default would have masked unbounded table growth. CleanupTask was already wired into the scheduler; no integration changes required (PR #133, bd-dmu8w).
- Stats tab no longer crashes with React error #185 (Maximum update depth exceeded) on every load. Recharts 3.x bundles `react-redux` + `@reduxjs/toolkit` for internal state; under React 19's stricter concurrent rendering, the chart's `YAxis` → `RenderedTicksReporter` `useEffect` re-dispatched `setRenderedTicks` with a fresh `ticks` array reference on every render, which forced a re-render that re-ran the effect, looping until React aborted with #185 inside the ErrorBoundary scoped to the Stats tab. The trigger was the React 18.3.1 → 19.2.5 bump in `ca6e5019` (bd-in620, 2026-04-24); the recharts 3.7.0 → 3.8.1 bump in `62ea5c5f` (2026-04-16) seeded `RenderedTicksReporter` itself, but the loop only manifested under React 19. Pin `recharts` to `2.15.4` — the latest 2.x line, declares React 19 support in peerDeps, and pre-dates the redux refactor entirely. Our chart usage (`LineChart`/`AreaChart`/`BarChart` + `XAxis`/`YAxis` + `Tooltip` + `ReferenceLine` + `ResponsiveContainer`) is all stable across the 2.x→3.x boundary; tsc clean and 110 stats panel tests pass against 2.15.4 (bd-a7l1z).

### Security
- `backend/routers/backup.py` — canonicalize-and-verify (`resolve()` + `relative_to()`) applied to `download_saved_backup` and `delete_saved_backup` as defense-in-depth alongside the existing `_BACKUP_FILENAME_RE` guard. Closes CodeQL py/path-injection alerts 1416-1419 (CWE-22/23/36/73/99). Pattern mirrors the zip-restore remediation in the same file. New tests cover traversal, absolute paths, symlink-escape, and null bytes (PR #98, bd-0a1pr).
- `backend/cloud_storage/onedrive_adapter.py` — Pydantic + adapter-level validators on `tenant_id` (GUID or verified domain) and `drive_id` (base64url-style) prevent SSRF via Microsoft Graph URL interpolation. Closes CodeQL CRITICAL py/partial-ssrf alerts 1361, 1362 (CWE-918). New tests at `backend/tests/test_cloud_storage.py` cover validator unit coverage + HTTP 422 at the cloud-target API (PR #101, bd-zbt74).
- `backend/auto_creation_schema.py` — three write-time `re.compile(<user_input>)` sites (condition regex validation, set_variable pattern validation, name_transform_pattern validation) now route through `safe_regex.compile`, which enforces a length cap and uniform compile-error surface. User-controlled regex from auto-creation rule definitions can no longer pin a worker via crafted ReDoS patterns. The bd-91mcq auto-creation bulk merges expanded request volume through these paths; this hardening closes the exposure window. Removes the three `# nosemgrep: no-bare-re-on-dynamic-pattern` annotations + their `TODO(enhancedchannelmanager-ltjyx)` markers. Backward-compatible: error message substrings (`"Invalid regex"`, `"Invalid name_transform_pattern"`) preserved for existing test/UI assertions. New `TestSafeRegexMigrationWriteTimeValidation` (10 tests) covers oversize-pattern rejection at all three sites + boundary-at-cap acceptance (PR #136, bd-ltjyx).
- `backend/routers/m3u_digest.py` — two write-time `re.compile(<user_input>)` sites validating `exclude_group_patterns` and `exclude_stream_patterns` from request bodies now route through `safe_regex.compile`. Same ReDoS class as bd-ltjyx, different surface (M3U digest endpoint instead of auto-creation rule endpoint). Removes the two `# nosemgrep` annotations. New tests in `TestUpdateDigestSettings` cover oversize-pattern rejection on both fields plus a `TestRouterUsesSafeRegex` source-level revert guard (PR #135, bd-3u6p0).
- Eight CodeQL py/stack-trace-exposure HIGH alerts (1412-1415, 1350-1353) remediated across `backend/routers/{backup,channels,export,health,settings}.py` + `backend/cloud_storage/`. Pattern: `logger.exception(...)` for full trace correlated by `X-Request-ID` + return only the exception class name (`type(e).__name__`) in the API response — never `str(e)` (which can include connection strings, file paths, or credentials). For `ImportError` paths in cloud_storage adapters, `e.name` (the missing module name) is used so the response stays useful without leaking interpreter paths. **Material risk closed**: alert 1414 in `health.py` — `/api/health/ready` is in `AUTH_EXEMPT_PATHS`, meaning DBAPI exception messages were echoed unauthenticated; this PR closes that. ADR-005 Phase 1 disallows "won't fix" dismissal so every open stack-trace alert needed a real fix. Six new sanitization tests + 1 updated, each asserting both the positive contract (response equals exception class) and the negative contract (the secret-bearing message body MUST NOT appear in the response) (PR #160, bd-m8i9q).
- `backend/export_manager.py` + `backend/routers/export.py` — defense-in-depth canonicalize-and-verify backstop for path traversal in the M3U/XMLTV export download paths. New `_safe_export_path(profile_id, *parts)` helper resolves and verifies containment under `EXPORTS_DIR.resolve()` via `Path.relative_to()`, raising `ExportPathError` on escape. Wired into `ExportManager.get_export_path` and `ExportManager.cleanup` (cleanup short-circuits before `rmtree` on escape). Also re-verifies path containment in `download_m3u`/`download_xmltv` before serving. CodeQL py/path-injection alerts 1354-1359 were already dismissed as false-positive (FastAPI int path-param + Pydantic `FILENAME_RE` allowlist make traversal unreachable upstream), but per ADR-005 sequencing 2 and the bd-0a1pr precedent the property is now guaranteed at the sink rather than depending on caller hygiene. New `TestExportPathInjection` (9 tests) mirrors `TestSavedBackupsPathInjection` from bd-0a1pr — traversal payloads, absolute paths, null-byte, backslash, cleanup sentinel-file regression, symlink-escape (PR #159, bd-h5rfv).

## [0.15.1] — 2026-03-31

### Added

- Injection / SSRF (OWASP A03/A10):
  - ffmpeg protocol allow-list on all invocations — blocks `file://`, `data://`, `concat:` and other dangerous protocols.
  - URL-scheme validation at input time on M3U, Xtream Codes, and EPG source endpoints — only `http://` and `https://` allowed.
  - Zip backup path canonicalization — defense-in-depth `resolve()` check alongside existing traversal guards.
- Logging hardening (OWASP A09):
  - 500-response scrubbing — global handler returns generic `Internal server error` to clients; real error stays in server logs.
  - Validation-error log redaction — `Authorization` / `Cookie` headers and auth-endpoint request bodies redacted from logs.

## [0.15.0] — 2026-03-29

### Added
- **Export & Publish** — generate M3U/XMLTV playlists from channel profiles and publish to cloud targets (S3, WebDAV, etc.) on a cron schedule. Includes channel selector, playlist preview, cloud-target management, and publish history.
